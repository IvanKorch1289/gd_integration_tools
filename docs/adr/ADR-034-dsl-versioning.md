# ADR-034: DSL apiVersion + migration framework

- **Статус:** accepted
- **Дата:** 2026-04-30
- **Фаза:** Wave-F-W25.3
- **Автор:** crazyivan1289

## Контекст

W25 вводит двусторонний канал YAML ↔ Python для DSL-маршрутов:

- watchdog hot-reload подхватывает `dsl_routes/*.yaml` в runtime;
- `RouteBuilder.to_spec()` пишет Pipeline обратно в YAML;
- `dsl_snapshots` (PostgreSQL) хранит исторические версии маршрутов.

Это значит, что spec-файлы могут жить долго: какие-то писались до
введения нового формата, какие-то будут писаться следующими волнами.
Без явного дискриминатора формата мы рискуем:

1. потерять обратную совместимость со старыми YAML и снапшотами при
   введении breaking change'а;
2. не понимать, какой формат лежит в БД, когда сравниваем версии;
3. производить миграции вручную, что приведёт к расхождениям между
   тем, что в YAML, и тем, что в Python-builder'е.

## Рассмотренные варианты

- **Вариант 1 — Schema-only (Pydantic).** Версионировать через
  Pydantic-модели (``PipelineSpecV0``, ``V1``, ``V2``). Каждый загрузчик
  выбирает модель по ``apiVersion`` и валидирует. Плюсы: строгая
  типизация. Минусы: для каждой версии нужна полная модель — boilerplate;
  миграция = ручной маппинг между моделями (то же, что и в
  framework'е, плюс дополнительный слой моделей); тяжело применить
  к снапшотам в БД, где формат уже как dict в JSONB.

- **Вариант 2 — Migration framework (выбран).** ``apiVersion`` —
  строковый дискриминатор; ``DSLMigration`` — Protocol с парой
  ``(from_version, to_version)``; ``MigrationRegistry`` — граф рёбер;
  ``apply_migrations(spec, target)`` — BFS по графу + последовательное
  применение. Плюсы: миграции — обычные dict→dict функции, легко
  реиспользуются в YAML loader, в alembic-снапшотах, в CLI; не требует
  модели на каждую версию; легко добавляется новая версия и легко
  удаляется устаревшая (когда все snapshot'ы и YAML обновлены). Минусы:
  типизации меньше — spec — это `dict[str, Any]`.

- **Вариант 3 — External migrator (cli only).** Хранить YAML в одной
  каноничной версии, миграции применять отдельным CLI-инструментом
  и не делать auto-upgrade в loader'е. Плюсы: loader проще. Минусы:
  старые YAML файлы (например, vendoring legacy) или импортированные
  снапшоты падают на parse — нужна координация между релизом и
  применением миграций; loader не self-healing.

## Решение

Выбираем **Вариант 2** — migration framework с auto-upgrade в
``load_pipeline_from_yaml`` и в ``PipelineVersionManager.compare``.

Демо-миграции v0→v1 и v1→v2 — identity (только маркируют историю
``_migrated_from``). Они нужны, чтобы:

1. обкатать API (registry, find_path, apply) на realistic-цепочке;
2. убедиться, что loader/CLI/snapshots правильно интегрированы;
3. оставить framework готовым к первому реальному breaking-change'у.

``CURRENT_VERSION`` фиксируется в ``src/dsl/versioning/migrations.py`` —
смена этого значения требует одновременного обновления:

- alembic-миграции `dsl_snapshots.api_version` (server_default);
- регистрации новой DSLMigration;
- юнит-тестов и публикации в `docs/reference/dsl/versioning.md`.

## Последствия

- **Положительные:**
  - Старые YAML загружаются автоматически без ручных правок.
  - Сравнение снапшотов из разных версий корректно (`compare`
    мигрирует к ``CURRENT_VERSION`` перед diff'ом).
  - Дальнейшая эволюция формата изолирована: одна миграция = одна
    функция ``dict -> dict``.

- **Отрицательные:**
  - Дополнительный шаг в loader'е — небольшое замедление при загрузке
    (не значимо: миграции — копирование ~10 ключей словаря).
  - Spec'и без ``apiVersion`` считаются v0 по умолчанию — может
    конфликтовать, если в проекте появится другой формат с тем же
    отсутствием поля. Mitigation: всегда писать ``apiVersion`` явно
    в новых файлах (Pipeline.to_dict делает это).

- **Нейтральные:**
  - Две колонки версии в `dsl_snapshots` (`version` для номера
    snapshot'а в истории + `api_version` для формата) — изначально
    путаница, но названия достаточно различные.
