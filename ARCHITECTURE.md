# ARCHITECTURE.md

## Назначение проекта

Проект — интеграционная шина на Python 3.14+ с DSL, workflow/orchestration, коннекторами, трансформациями и developer portal.

Цель архитектуры:
- расширять функциональность без переписывания ядра;
- описывать интеграции декларативно через DSL;
- минимизировать связность между слоями;
- обеспечивать безопасное и предсказуемое внесение изменений.

## Архитектурная схема

```text
entrypoints/ -> services/ -> infrastructure/
     |              |             |
  schemas/      core/di      core/protocols
               core/dsl      core/interfaces
```

## Границы слоёв

- `entrypoints` импортирует только `services`, `schemas`
- `services` импортирует только `core`, `schemas`
- `infrastructure` реализует контракты из `core`
- `core` не импортирует код из `src/`

## Основные подсистемы

### 1. DSL
Назначение:
- декларативное описание интеграций, маршрутизации, трансформаций и workflow.

Ключевые точки:
- `src/dsl/`
- `src/core/service_dsl.py` или аналогичные модули
- регистрация DSL-возможностей в сервисном слое

### 2. Workflow / orchestration
Назначение:
- описание и выполнение бизнес-процессов, ветвлений, циклов и subworkflows.

Ключевые точки:
- `src/workflows/`
- `src/infrastructure/workflow_engine.py`
- интеграция с DSL и DI

### 3. Коннекторы
Назначение:
- подключение внешних транспортов и систем: REST / SOAP / gRPC / GraphQL / Queue.

Ключевые точки:
- `src/infrastructure/connectors/`
- `src/core/protocols.py`
- `src/core/interfaces.py`

### 4. Сервисный слой
Назначение:
- прикладная логика, orchestration use cases, координация контрактов и инфраструктуры.

Ключевые точки:
- `src/services/`

### 5. Схемы и контракты
Назначение:
- стабилизация API, payload, внутренних контрактов и форматов обмена.

Ключевые точки:
- `src/schemas/`
- `src/core/protocols.py`
- `src/core/interfaces.py`

### 6. Developer portal / UI
Назначение:
- навигация по проекту, визуализация workflow, документация, вспомогательные инструменты.

Ключевые точки:
- Streamlit / portal-модули
- связанные статические или docs-компоненты

## Инфраструктурные зависимости

Используются или могут использоваться:
- БД
- Redis
- RabbitMQ / очереди
- внешние API
- файловые/сетевые интеграции

Для каждой зависимости важно:
- как создаётся подключение;
- через какой интерфейс она используется;
- где проходит граница между core и infrastructure.

## DI и точки расширения

Все расширения должны по возможности проходить через:
- протоколы;
- интерфейсы;
- DI-регистрацию;
- декларативное описание в DSL.

Нежелательно:
- прямое создание инфраструктурных реализаций в сервисах;
- хардкод конфигурации;
- bypass DI.

## Поток данных

Типовой путь:
1. Вход через entrypoint
2. Валидация / схемы
3. Вызов service
4. Использование core-контрактов
5. Делегирование в infrastructure
6. Возврат результата / публикация события / запуск workflow

## Правила изменения архитектуры

Перед изменениями:
- смотреть Graphify;
- проверять затронутые импортёры;
- учитывать DSL-покрытие;
- учитывать DI-регистрацию;
- проверять влияние на contracts/schemas/routes/actions.

## Известные ограничения

- Архитектурная правда — в коде, Graphify и актуальных архитектурных документах.
- `ARCHITECTURE.md` — обзорный документ, его нужно обновлять при существенных изменениях.
- Если обзор расходится с кодом — доверять коду и Graphify.

## Что обновлять при крупных изменениях

Обновляй `ARCHITECTURE.md`, если:
- появилась новая крупная подсистема;
- изменились границы слоёв;
- появился новый тип коннектора;
- изменился workflow-engine;
- изменился основной способ DSL-расширения;
- изменилась карта инфраструктурных зависимостей.
