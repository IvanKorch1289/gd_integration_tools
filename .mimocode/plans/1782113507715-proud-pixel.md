## A. EXECUTIVE SUMMARY

### 10–20 Key Findings

1. **CRITICAL: YAML watcher has no caching** — Full rescan + reparse on every FS event (triple filesystem traversal per reload)
2. **CRITICAL: Blocking `threading.Lock` in async `EmbeddingVectorCache`** — Blocks event loop, found at `infrastructure/cache/rag/embedding_cache.py:19,35,53`
3. **CRITICAL: Per-request HTTP client creation** — DSL processors create `HttpClient()` per request instead of singleton reuse
4. **CRITICAL: API Key backend weak hashing** — SHA-256 without salt at `core/auth/api_key_backend.py:17-20`
5. **CRITICAL: Extensions violate layer boundaries** — `osint_agent` imports from `infrastructure/` directly
6. **HIGH: FileSink has no path traversal protection** — `path` parameter accepted directly without sanitization
7. **HIGH: SSE endpoints have no authentication** — `GET /events/stream` and `POST /events/invoke` bypass auth
8. **HIGH: WebSocket auth is optional** — `WSAuthenticator` not enforced on connection
9. **HIGH: CDC PostgreSQL logical replication is scaffold** — Real implementation deferred to Wave R3
10. **HIGH: HITL uses busy-wait polling** — `_wait_for_decision()` polls instead of event-driven wakeup
11. **HIGH: pg_runner_backend uses busy-wait polling** — No LISTEN/NOTIFY push mechanism
12. **HIGH: 200+ processor types in core** — Domain-specific processors (Citrix, TerminalEmulator, KycAml) should be extensions
13. **MEDIUM: Frontend is NOT a thin client** — 39 direct `src.backend` imports, pages execute backend logic
14. **MEDIUM: Complex RouteBuilder MRO** — 30+ mixins creating diamond problem potential
15. **MEDIUM: No pipeline-level backpressure** — Only in streaming consumers
16. **MEDIUM: Dual bulkhead implementations** — `core/resilience/bulkhead.py` vs `infrastructure/resilience/bulkhead.py`
17. **MEDIUM: SAML replay store in-memory** — Lost on restart, no horizontal scaling
18. **MEDIUM: Policy checked before declaration** — Could grant undeclared capabilities
19. **MEDIUM: SupervisorProcessor not implemented** — Only stores config, no restart logic
20. **LOW: Duplicate testkit packages** — `testkit/` and `src/testkit/` both exist

## Summary

The repository demonstrates **high architectural sophistication** with a well-designed core, comprehensive DSL, and multi-backend workflow support. However, several **critical security and performance issues** must be addressed before scaling:

1. **Security**: FileSink path traversal, SQL injection in event_log, extension layer violations, weak API key hashing
2. **Performance**: YAML watcher no caching, per-request HTTP clients, busy-wait polling
3. **Agent Safety**: Tool policy not enforced, InProcessAgentSandbox has zero isolation
4. **Architecture**: Complex RouteBuilder MRO, processor proliferation, dual bulkhead implementations

The refactoring roadmap prioritizes **quick security fixes** (Horizon 1), **layer boundary corrections** (Horizon 2), and **platform evolution** (Horizon 3) with careful attention to not breaking already-working parts of the system.

**Estimated Total Effort**: 3-4 weeks for Horizon 1+2, 1-3 months for Horizon 3
