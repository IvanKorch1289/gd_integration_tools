## АРХИТЕКТУРНЫЕ ПРАВИЛА
Применяй как абсолютные ограничения. Любое нарушение — дефект.

ПРАВИЛО 1 — ПРИНЦИП ФАСАДА
Любая сквозная capability (storage, cache, уведомления, события,
секреты, auth, внешний HTTP, внешние БД, логирование, аудит) ДОЛЖНА
использоваться через единый интерфейс-фасад. Потребитель не должен
знать о бэкенде. Пример: NotificationFacade.send() сам маршрутизирует
в Email / Telegram / Push в зависимости от конфига — прямой импорт
бэкенда из services/ или dsl/ запрещён.
→ Аудит: перечисли каждый прямой импорт бэкенда в services/ или dsl/,
  обходящий фасад. Каждый — нарушение.

ПРАВИЛО 2 — НЕЗАВИСИМОСТЬ СЛОЁВ
Допустимые направления импорта ТОЛЬКО:
  entrypoints → services → core/interfaces
  dsl         → core/interfaces
  infrastructure реализует core/interfaces
  services    → infrastructure (только через DI/фасад, никогда напрямую)
  plugins     → core/interfaces (никогда во внутренние infrastructure)
→ Аудит: найди sibling-layer imports, circular imports, cross-imports
  service→dsl. Укажи файл + строку для каждого.

ПРАВИЛО 3 — НЕТ МЁРТВОМУ КОДУ / ОВЕРИНЖИНИРИНГУ
Функция/класс — мёртвый, если: никогда не импортируется, нет теста,
содержит TODO без реализации, является scaffold-заглушкой без логики.
Паттерн — оверинжиниринг, если существует кастомная реализация там,
где хорошо поддерживаемая библиотека покрывает тот же случай точно.
→ Для каждой находки мёртвого кода: удалить или реализовать в текущем спринте.
→ Для каждого оверинжиниринга: указать библиотеку-замену +
  оценку усилий (S/M/L).

ПРАВИЛО 4 — БИБЛИОТЕКИ ВМЕСТО КАСТОМНОГО КОДА
Аудируй конкретные кандидаты на замену:
  - Circuit Breaker      → оставить только purgatory, удалить кастомные
  - Rate Limiter         → оставить только limits, удалить остальные
  - Retry/backoff        → tenacity
  - DI-контейнер         → оценить dishka vs текущий svcs (async-поддержка)
  - Structured LLM output → instructor
  - AI guardrails        → guardrails-ai или nemo-guardrails
  - Agent memory         → mem0ai
  - Prompt caching       → Anthropic ephemeral cache API
  - Конфиг/константы     → dynaconf с Consul-источником
  - Генерация документации → mkdocs-material + mkdocstrings
  - Планировщик          → APScheduler 4.x или rocketry
  - Валидация данных      → annotated-types + msgspec там, где Pydantic избыточен
  - Streaming LLM        → langchain-core streaming или litellm
  - Chaos engineering    → chaostoolkit вместо самописного модуля

ПРАВИЛО 5 — ПОЛНОТА DSL
Каждая инфраструктурная capability, подключённая end-to-end, ДОЛЖНА
иметь хотя бы один метод RouteBuilder, вызываемый в fluent-стиле:
  route.to_eventbus("topic")
  route.from_db(query=...).to_cache(ttl=60)

→ Аудит: для каждой capability проверь: (а) Processor существует,

  умолчанию или задокументирована причина отключения. Перечисли все пробелы.

ПРАВИЛО 6 — УСТОЙЧИВОСТЬ ИНФРАСТРУКТУРЫ
Каждое внешнее подключение ДОЛЖНО иметь:

→ Аудит: для каждого клиента (HTTP, Redis, S3, PostgreSQL, внешняя БД,

ПРАВИЛО 9 — PYTHON 3.14: МАКСИМАЛЬНОЕ ИСПОЛЬЗОВАНИЕ
Проект должен использовать возможности Python 3.14 и обходить его слабые
стороны. Применяй это правило ко всем рекомендациям.
  Используй:
  - PEP 696 (TypeVar defaults), PEP 702 (deprecated decorator),
    PEP 742 (TypeIs), PEP 749 (lazy annotations)
  - annotated-types + typing.TypeAliasType для самодокументируемых типов
  - t.override для безопасного переопределения методов
  - match/case с полным structural pattern matching вместо if/elif цепочек
  - asyncio.TaskGroup вместо gather() для структурированного параллелизма
  - ExceptionGroup + except* для агрегирования ошибок в async-коде
  - importlib.resources вместо __file__-path hacks
  - tomllib для чтения конфигов без зависимостей
  - pathlib везде вместо os.path
  - dataclasses с slots=True, kw_only=True для value objects
  - __slots__ в горячих классах для экономии памяти
  - PEP 695 type aliases (type Vector = list[float]) вместо TypeAlias
  Обходи:
  - GIL (по умолчанию активен) → для CPU-bound задач использовать
    ProcessPoolExecutor или Dask, а не threading
  - asyncio.gather() без TaskGroup теряет ошибки — заменить на TaskGroup
  - Глубокая рекурсия → itertools / explicit stack
  - import time — lazy imports через importlib для тяжёлых зависимостей
  - Память — избегать накопления объектов в event loop; использовать
    __slots__, weakref, генераторы вместо списков где возможно
→ Аудит: найди код, использующий устаревшие паттерны Python < 3.12,
  который можно улучшить. Укажи файл + строку + замену.

ПРАВИЛО 10 — АГЕНТСКАЯ ИЗОЛЯЦИЯ
Агентские workflow (LangGraph и любой LLM-агент) ДОЛЖНЫ:
  - Исполняться в изолированном окружении (E2B sandbox или отдельный
    subprocess), никогда в main event loop
  - Получать ТОЛЬКО явно переданный контекст — без доступа к файловой
    системе проекта, переменным окружения, внутренним сервисам
  - Иметь строгие input/output схемы (Pydantic) с валидацией на входе
    и выходе; отклонять output, не соответствующий схеме
  - Ограничения инструментов (tool whitelist/blacklist) применяться через
    OPA/Rego или guardrails-ai, НЕ через markdown-файл
  - Иметь timeout + token budget + cost budget; превышение = принудительная
    остановка, не мягкое предупреждение
  - Все вызовы инструментов логироваться с трассировкой через OpenTelemetry
→ Аудит: проверь каждый агентский граф. Укажи места, где агент имеет
  доступ к внутренним сервисам без явной изоляции.

ПРАВИЛО 11 — FASTAPI / ENTRYPOINTS: ПОЛНОТА И БЕЗОПАСНОСТЬ
  - Все роуты документированы через OpenAPI (summary, description, tags)
  - Файловая передача поддерживает: multipart upload, streaming download,
    presigned URL, base64 fallback, сжатие (gzip/zstd)
  - gRPC эндпоинты имеют .proto с bytes-полями для бинарных данных
  - WebSocket-эндпоинты имеют heartbeat, reconnect-логику, graceful close
  - Rate limiting применяется на уровне middleware, не inline в роутах
  - Все аутентификационные механизмы (JWT, API key, OAuth2) идут через
    единый AuthFacade, а не через прямые зависимости
→ Аудит: найди роуты без документации, без rate limiting, с прямым

ПРАВИЛО 12 — ПРОИЗВОДИТЕЛЬНОСТЬ И ПАРАЛЛЕЛИЗМ
  - Все I/O операции — async; никакого блокирующего I/O в event loop
  - Batch-операции для БД: insert/update/delete пачками, не поштучно
  - Кэширование на уровне фасада: TTL + стратегия инвалидации задокументированы
  - CPU-bound задачи (парсинг, шифрование, ML inference) выносятся в

  - Streaming: large file transfer и large dataset processing через async

  - Connection pool size настраивается через Settings, не хардкодится
