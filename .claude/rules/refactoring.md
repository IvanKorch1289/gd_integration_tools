# Правила рефакторинга и разработки

## Язык кода
- Все докстринги и комментарии — строго на русском языке
- Код (переменные, функции, классы) — на английском

## Python 3.14
- Использовать максимальную функциональность Python 3.14
- match/case вместо if/elif цепочек
- `TypeAlias`, PEP 695 type parameters, `@override`
- `tomllib`, `pathlib`, walrus-оператор, `asyncio.TaskGroup`
- `from __future__ import annotations` при необходимости

## Принципы упрощения
- НЕ писать кастомные функции, если есть библиотечные аналоги
- Перед написанием функции — искать аналоги в stdlib и библиотеках проекта
- Если полезная библиотека не подключена — подключить с обоснованием
- Удалять мёртвый код, лишние импорты, неиспользуемые зависимости
- Заменять устаревшие библиотеки современными аналогами
- Тяжёлые библиотеки — заменять на лёгкие, если нет деградации качества

## Архитектурные правила
- Максимальная абстракция: протоколы → интерфейсы → реализации
- Функции с похожей логикой — объединять
- Сложные функции/файлы — разбивать
- Файлы — организовывать по интуитивно понятным директориям
- Приоритет: отказоустойчивость → скорость → читаемость

## Запрещённые паттерны
- God Object (класс >300 строк или >10 публичных методов)
- Прямой импорт infrastructure в services/core
- Хардкод конфигурации и секретов
- `import time; time.sleep()` в async-контексте
- Прямой `SomeClass()` в обход DI-контейнера
- `except Exception: pass` (глотание ошибок)
- Логирование через print или logging.basicConfig

## Рефакторинг модуля (порядок шагов)
```
1. graphify explain "<модуль>"
2. make test           ← зафиксировать baseline
3. Внести изменение (1 шаг)
4. make test           ← поведение не сломалось?
5. make lint           ← ruff + mypy
6. graphify update .   ← обновить граф
7. git commit -m "refactor: ..."
```

## Именование
- Классы: PascalCase; сервисы: глагол+сущ (OrderProcessor, не Orders)
- Методы: snake_case, глагол+сущ (process_order)
- Приватные: _prefix; Константы: UPPER_SNAKE_CASE
- Файлы: snake_case.py

## DSL
- Дальнейшее развитие DSL — покрывать им все возможности проекта
- За основу — популярные интеграционные фреймворки (Apache Camel, MuleSoft)
- DSL должен описывать: коннекторы, трансформации, роутинг, workflow-оркестрацию
- Цель: минимум изменений ядра при добавлении новой функциональности

## Workflow
- Workflow поддерживают: бизнес-логику, ветвление, циклы, subworkflows
- Workflow запускаются в отдельном потоке/процессе
- Приоритет: использовать готовый workflow-движок (Temporal, Prefect, Celery) вместо написания своего
- Визуализация: Streamlit или developer portal

## Бизнес-функции
- Универсальные: подключаются ко всем коннекторам при создании
- Поддерживают вызов через: REST / SOAP / gRPC / GraphQL / Queue
- Режимы ответа: синхронный / отложенный / фоновый

## Фронт
- Только Streamlit, лёгкий и интуитивный
- Чёткая навигация: dashboard, workflow-builder, developer portal
