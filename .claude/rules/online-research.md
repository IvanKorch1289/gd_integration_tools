# Политика внешнего исследования

## Когда идти в интернет
Использовать web search / web fetch только если задача требует:
- актуальной документации библиотек;
- release notes / changelog;
- breaking changes;
- новых паттернов разработки;
- свежих примеров интеграции;
- сравнения современных инструментов.

## Доступные инструменты
- **Context7 MCP** (`context7`) — **первичный** источник для документации библиотек, frameworks, SDK, API, CLI tools, cloud services. Версионная актуальная документация прямо из официальных репозиториев. Использовать ВСЕГДА для вопросов о библиотеках, даже хорошо известных (FastAPI, Pydantic, SQLAlchemy, Temporal, etc.) — training data может не отражать свежие изменения.
  - `mcp__context7__resolve-library-id` — найти canonical library-id по имени.
  - `mcp__context7__query-docs` — получить актуальный фрагмент документации по библиотеке + теме.
- **DuckDuckGo MCP** (`duckduckgo` / `duckduckgo-search`) — общий веб-поиск, когда Context7 не покрывает (новости, blog-посты, сравнения, статус проекта, GitHub issues, миграционный опыт).
- **Fetch MCP** (`fetch`) — официальный Anthropic-сервер для углублённого чтения 1–2 страниц после поиска.
- **WebSearch / WebFetch** — встроенные инструменты Claude Code как fallback (если MCP недоступен).

## Когда какой инструмент
| Вопрос | Инструмент |
|---|---|
| «Как API библиотеки X работает», «какие методы у класса Y», «синтаксис конфигурации Z» | **Context7** (без альтернатив) |
| «Версия библиотеки X совместима с Python 3.14», «breaking changes между X.5 и X.6» | **Context7** → официальная документация версии; fallback DuckDuckGo по `release notes`/`changelog` |
| «Что лучше: библиотека A или B», «есть ли современная замена X» | **DuckDuckGo** (сравнения, blog-посты) + **Context7** для каждого кандидата |
| «Известная проблема X с Y», «миграционный опыт сообщества» | **DuckDuckGo** + GitHub issues |
| «Свежий best practice по теме X» | **DuckDuckGo** + **Fetch** на топ-1 источник + **Context7** для контракта |
| «Статус проекта: active/archived/deprecated» | **DuckDuckGo** (commit recency, last release) |

## Порядок
1. Сначала проверить внутренний контекст проекта через Graphify и docs.
2. Затем коротко сформулировать, чего именно не хватает.
3. Если вопрос про API/конфигурацию/синтаксис библиотеки — начать с **Context7** (`resolve-library-id` → `query-docs`).
4. Если вопрос про сравнение/статус/мнения — начать с **DuckDuckGo MCP** узким запросом.
5. При необходимости использовать Fetch MCP / WebFetch только для 1–2 наиболее релевантных источников.
6. В ответе явно отделять:
   - внутренний контекст проекта;
   - внешние актуальные данные (с источником и датой, если применимо);
   - Context7-цитаты помечать `[ctx7: <library>@<version>]`.

## Источники по умолчанию
Предпочитать:
- официальную документацию;
- GitHub releases / changelog;
- блоги вендоров;
- авторитетные engineering-статьи.

Избегать:
- случайных SEO-статей;
- устаревших gist без контекста;
- анонимных непроверяемых источников.

## Если поиск недоступен
Явно сообщить пользователю: «ответ без актуальной внешней проверки» — не маскировать ограничение.
