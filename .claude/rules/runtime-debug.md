# Правила диагностики runtime и интеграций

## Когда использовать
- ошибка запуска;
- нестабильное поведение интеграций;
- проблемы с БД / Redis / RabbitMQ;
- зависания, таймауты, утечки памяти;
- проблема воспроизводится только на локальной инфраструктуре.

## Обязательный порядок
1. Сначала воспроизведи симптом минимальной командой.
2. Не читай весь проект — только entrypoint, связанный service, конфигурирование DI, инфраструктурный адаптер.
3. Если проблема связана с HTTP/API — проверь `make routes`, `make actions`, OpenAPI и соответствующий router.
4. Если проблема связана с окружением — проверь только безопасные файлы конфигурации, без чтения `.env`.
5. Если проблема связана с производительностью или памятью — предложи `make profile-memray`, `make profile-mprof` или `make profile-pyspy`.

## Что проверять в первую очередь
- нарушение DI-регистрации;
- несовпадение схем и runtime-конфигурации;
- неправильный async/sync boundary;
- сетевые retry/timeout;
- блокировки на Redis/Rabbit/БД;
- скрытые broad exception handlers;
- расхождение DSL-описания и фактического рантайма.

## Формат ответа
1. Симптом
2. Наиболее вероятные причины
3. Какие файлы читать точечно
4. Какая одна команда лучше всего подтверждает гипотезу
5. Самое маленькое безопасное исправление
